# Framework Workshop Facilitation Guide

This guide provides facilitation patterns for different types of frameworks. Use this to understand how to adapt your workshop style based on framework type.

---

## General Workshop Structure

All framework workshops follow this basic pattern, adapted to framework type:

### Phase 1: Setup & Context (5-10 min)
- Confirm framework and load definition
- Understand situation context
- Set workshop expectations
- Answer clarifying questions

### Phase 2: Framework Introduction (5-10 min)
- Explain framework origin and purpose
- Present framework structure/phases
- Provide relevant example
- Clarify before diving in

### Phase 3: Guided Application (20-40 min)
- Work through each framework phase systematically
- Ask facilitation questions
- Probe for depth
- Connect insights across phases
- Flag common pitfalls

### Phase 4: Analysis & Insights (5-10 min)
- Synthesize findings
- Identify patterns and themes
- Generate prioritized recommendations
- Define next steps
- Document in framework-workshop.md

### Phase 5: Wrap-up (2-5 min)
- Summarize key takeaways
- Encourage action
- Offer follow-up options

## Framework Type 1: Strategic Frameworks

**Frameworks:** Porter's Five Forces, Blue Ocean Strategy, Wardley Mapping, SWOT Analysis, BCG Matrix, Ansoff Matrix, McKinsey 7S, Scenario Planning, Business Model Canvas, Value Proposition Canvas, Lean Canvas

### Characteristics
- Focus on competitive positioning and market dynamics
- Analytical and data-driven
- External environment focused
- Long-term orientation
- Multiple stakeholders

### Facilitation Style: Analytical & Thorough

**Tone:**
- Professional and business-focused
- Analytical but not dry
- Challenge assumptions rigorously
- Connect analysis to strategic choices

**Pacing:**
- Methodical and complete
- Don't rush analysis phases
- Allow time for deep thinking
- Balance thoroughness with momentum

**Key Behaviors:**

**1. Push for Evidence:**
- "What data supports that?"
- "How do you know the market is [X]?"
- "Can you quantify that force/factor?"

**2. Challenge Optimism:**
- "Many companies think that. What makes you different?"
- "If this opportunity is so good, why isn't everyone doing it?"
- "What's the counter-argument?"

**3. Force Prioritization:**
- "Of these five forces, which two matter most?"
- "Which strategic option has the most leverage?"
- "What's the single biggest factor?"

**4. Connect to Action:**
- Strategic frameworks can lead to analysis paralysis
- Always end with: "Based on this analysis, what changes?"
- "What specific strategic moves follow from this?"

**5. Test Assumptions:**
- "You're assuming [X]. What if that's wrong?"
- "This analysis holds if [assumption]. How confident are you?"
- "What would need to be true for this to fail?"

### Common Pitfalls to Avoid

⚠️ **Analysis paralysis** - Too much thinking, no decisions
- Solution: Set time limits, force prioritization, demand next steps

⚠️ **Ignoring execution** - Great strategy, impossible implementation
- Solution: Ask "How would you actually do this?" throughout

⚠️ **Assuming static environment** - Analysis becomes outdated
- Solution: Build in "What's changing?" questions

⚠️ **Overconfidence in analysis** - Treating educated guesses as facts
- Solution: Mark assumptions vs validated facts explicitly

### Workshop Flow Example: Porter's Five Forces

**Setup (5 min):**
- "We're analyzing competitive forces in your industry"
- "We'll examine 5 forces: rivalry, new entrants, substitutes, suppliers, buyers"
- "End goal: Understand where you have power and where you're vulnerable"

**Introduction (5 min):**
- Michael Porter, 1979, Harvard
- Framework for industry structure analysis
- Example: "When Uber entered taxis, they changed the competitive rivalry force dramatically"

**Application (30 min):**
- **For each force:**
  1. Explain what this force means
  2. Ask specific questions about user's industry
  3. Rate force strength (high/medium/low)
  4. Identify evidence
  5. Connect to strategic implications

**Synthesis (10 min):**
- "Which forces are strongest/weakest?"
- "Where do you have power?"
- "What strategic moves does this suggest?"
- Document with clear recommendations

## Framework Type 2: Mental Models

**Frameworks:** Mental Models (Munger), First Principles Thinking, Second-Order Thinking, Inversion, Circle of Competence, Systems Thinking, Regret Minimization

### Characteristics
- Focus on thinking patterns and cognitive clarity
- Philosophical and abstract
- Broadly applicable
- Timeless principles
- Often personal/individual application

### Facilitation Style: Philosophical & Probing

**Tone:**
- Thoughtful and reflective
- Socratic questioning
- Challenge conventional wisdom
- Comfortable with big questions

**Pacing:**
- Allow pauses for reflection
- Don't rush insights
- Some mental models need time to "click"
- It's okay to sit with discomfort

**Key Behaviors:**

**1. Go Deep on Fundamentals:**
- "What's fundamentally true here?"
- "Strip away conventions - what remains?"
- "What would Elon Musk ask?" (First Principles)
- "What would Charlie Munger consider?" (Mental Models)

**2. Explore Multiple Perspectives:**
- "How would an economist see this?"
- "What would a biologist notice?"
- "What lens are you NOT using?"

**3. Challenge "Obvious" Answers:**
- "Everyone thinks that. What if they're wrong?"
- "That's conventional wisdom. What are the fundamental truths?"
- "You're reasoning by analogy. What are the first principles?"

**4. Use Thought Experiments:**
- "Imagine the opposite. What happens?"
- "If you had to start from zero, what would you do?"
- "Think 10 years out. Do you regret this?"

**5. Make Abstract Concrete:**
- Mental models can be too abstract
- Always ground in user's specific situation
- "Apply that principle to your situation: what changes?"

### Common Pitfalls to Avoid

⚠️ **Staying too abstract** - Philosophy without application
- Solution: Constantly connect to user's concrete situation

⚠️ **Overthinking simple decisions** - Using mental models for obvious choices
- Solution: "Is this decision complex enough to warrant this depth?"

⚠️ **Paralysis by analysis** - Seeing too many perspectives
- Solution: Force synthesis: "Given all perspectives, what's the best path?"

⚠️ **Intellectual show-off** - Citing models without adding value
- Solution: Focus on what's useful, not impressive

### Workshop Flow Example: First Principles Thinking

**Setup (5 min):**
- "We're going to break your problem down to fundamental truths"
- "Then rebuild from scratch, ignoring how things are 'supposed to be done'"

**Introduction (5 min):**
- Elon Musk's favorite framework
- Aristotle's ancient principle
- Example: "SpaceX questioned 'Why are rockets so expensive?' and rebuilt from physics"

**Application (25 min):**
1. **Define problem** (5 min)
   - What are you trying to achieve?
   
2. **List assumptions** (10 min)
   - What do you assume must be true?
   - Which are real constraints vs tradition?
   
3. **Identify first principles** (5 min)
   - What MUST be true? (physics, math, human nature)
   - What's just "how we've always done it"?
   
4. **Rebuild from first principles** (10 min)
   - Start only with fundamentals
   - How would you approach this fresh?

**Synthesis (8 min):**
- Compare current approach vs first principles approach
- What changes?
- What specific actions follow?

## Framework Type 3: Decision Frameworks

**Frameworks:** OODA Loop, Cynefin Framework, Pre-Mortem Analysis, Eisenhower Matrix, Pareto Principle, Kano Model, Risk Management Frameworks

### Characteristics
- Focus on making choices and managing options
- Pragmatic and action-oriented
- Trade-off analysis
- Risk assessment
- Clear decision outputs

### Facilitation Style: Pragmatic & Action-Focused

**Tone:**
- Clear and direct
- Action-oriented
- Decision-forcing
- Comfortable with trade-offs

**Pacing:**
- Brisk but not rushed
- Move toward decision
- Don't overanalyze
- "Good enough" > "perfect"

**Key Behaviors:**

**1. Force Choices:**
- "You have to choose. Which one?"
- "What's the decision here?"
- "If you had to decide right now, what would it be?"
- "Pick A or B. You can't have both."

**2. Explore Trade-offs:**
- "What do you give up if you choose this?"
- "What's the cost of this option?"
- "You can optimize for X or Y, not both. Which matters more?"

**3. Assess Risks Explicitly:**
- "What's the downside?"
- "What could go wrong?"
- "How reversible is this?"
- "What's your backup plan?"

**4. Challenge Indecision:**
- "What's stopping you from deciding?"
- "What information would change your mind?"
- "Are you overthinking this?"

**5. Push for Commitment:**
- Decision frameworks should end with decisions
- "So what are you going to do?"
- "When will you act on this?"
- "What's the first step?"

### Common Pitfalls to Avoid

⚠️ **Premature closure** - Deciding too fast without exploration
- Solution: Ensure options are fully understood before forcing choice

⚠️ **Confirmation bias** - Looking only for data supporting preferred option
- Solution: "What evidence would make you choose differently?"

⚠️ **Analysis paralysis** - Wanting more data before deciding
- Solution: "What's the cost of delay? Can you decide with what you know?"

⚠️ **Ignoring consequences** - Not thinking through second-order effects
- Solution: "Then what happens?" (repeat 3 times)

### Workshop Flow Example: Pre-Mortem Analysis

**Setup (5 min):**
- "We're going to imagine your plan failed spectacularly"
- "Then work backwards to figure out what went wrong"
- "This helps spot risks before they happen"

**Introduction (5 min):**
- Gary Klein, psychologist
- Based on prospective hindsight
- Example: "Before Amazon launched AWS, they imagined it failing - helped them avoid those failures"

**Application (25 min):**
1. **Describe plan/decision** (5 min)
   - What are you planning to do?
   
2. **Imagine failure** (5 min)
   - "It's 12 months from now. The project failed completely. What happened?"
   - Write failure story
   
3. **Identify causes** (10 min)
   - What led to failure?
   - List specific reasons
   - Go from obvious to subtle
   
4. **Assess likelihood & impact** (5 min)
   - Which failures are most likely?
   - Which would hurt most?

**Synthesis (8 min):**
- Prioritized list of risks
- Mitigation strategies for top risks
- Go/no-go decision
- What to monitor

## Framework Type 4: Innovation Frameworks

**Frameworks:** Design Thinking, Jobs-to-be-Done, Lean Startup, Six Thinking Hats

### Characteristics
- Focus on discovery and creation
- User/customer-centered
- Iterative and experimental
- Embrace uncertainty
- Creative and exploratory

### Facilitation Style: Creative & Exploratory

**Tone:**
- Open and curious
- Encourage wild ideas
- Defer judgment
- Playful but purposeful

**Pacing:**
- Faster in ideation
- Slower in understanding users
- Multiple iterations expected
- Embrace "messiness"

**Key Behaviors:**

**1. Champion User Perspective:**
- "What does the user actually need?"
- "What job are they trying to do?"
- "What frustrates them about current solutions?"
- "What do they really value?"

**2. Encourage Divergent Thinking:**
- "What else? And what else?"
- "Go weirder. What's a crazy solution?"
- "What would [unrelated industry] do?"
- "If you had unlimited resources, what would you build?"

**3. Defer Judgment:**
- During ideation: "No buts, only ands"
- "Let's capture all ideas first, evaluate later"
- "What's interesting about that idea?"

**4. Push for Prototypes:**
- "What's the fastest way to test this?"
- "How could you fake this?"
- "What's the minimum to test the hypothesis?"
- "Can you test this without building it?"

**5. Emphasize Learning Over Being Right:**
- "What did you learn?"
- "What surprised you?"
- "What would you test next?"
- "How did reality differ from expectations?"

### Common Pitfalls to Avoid

⚠️ **Skipping empathy/research** - Jumping to solutions
- Solution: Force user understanding first. "How do you KNOW users need this?"

⚠️ **Falling in love with solution** - Not testing, assuming it works
- Solution: "What would prove you wrong? Let's test that."

⚠️ **Overbuilding** - Making too much before testing
- Solution: "What's the absolute minimum to test your hypothesis?"

⚠️ **Ignoring business viability** - Great user experience, bad business
- Solution: "This solves user needs. But will they pay? Can you deliver profitably?"

### Workshop Flow Example: Design Thinking

**Setup (5 min):**
- "We're going through 5 phases: Empathize, Define, Ideate, Prototype, Test"
- "Focus is on understanding users deeply, then creating solutions"

**Introduction (5 min):**
- Stanford d.school, IDEO
- Human-centered innovation
- Example: "IDEO redesigned shopping carts by observing how people actually shop"

**Application (35 min):**
1. **Empathize** (10 min)
   - Who's the user?
   - What are they trying to do?
   - What frustrates them?
   - What do they really need (vs what they say)?
   
2. **Define** (5 min)
   - Point of View statement: "[User] needs [need] because [insight]"
   - "How Might We..." question
   
3. **Ideate** (10 min)
   - Brainstorm solutions (quantity over quality)
   - Go wild, defer judgment
   - Cluster and select top ideas
   
4. **Prototype** (5 min)
   - Pick one idea to prototype
   - What's the minimum viable prototype?
   - How to make it fast and cheap?
   
5. **Test** (5 min)
   - Who will test?
   - What questions to ask?
   - What to observe?
   - Learning plan

**Synthesis (8 min):**
- Selected concept to prototype
- Test plan
- Expected learnings
- Next iteration

## Framework Type 5: Operational Frameworks

**Frameworks:** Theory of Constraints, OKR Framework, Pareto Principle (80/20), Systems Thinking, OODA Loop

### Characteristics
- Focus on execution and optimization
- Metrics and measurement
- Process improvement
- Resource allocation
- Results-oriented

### Facilitation Style: Practical & Metric-Driven

**Tone:**
- Business-focused and pragmatic
- Numbers-oriented
- Efficiency-minded
- Results-focused

**Pacing:**
- Brisk and efficient
- Get to actionable quickly
- Measure and iterate
- Bias toward action

**Key Behaviors:**

**1. Focus on Bottlenecks:**
- "Where's the constraint?"
- "What's the limiting factor?"
- "If you could improve only one thing, what would have most impact?"

**2. Demand Metrics:**
- "How do you measure that?"
- "What's the current baseline?"
- "How will you know if it's working?"
- "What's good/great/excellent performance?"

**3. Force Prioritization:**
- "What's the 20% that drives 80% of results?"
- "You can't optimize everything. What matters most?"
- "What should you STOP doing?"

**4. Connect to Business Results:**
- "How does this affect revenue/cost/profit?"
- "What's the ROI?"
- "How does this move the business forward?"

**5. Push for Speed:**
- "How fast can you implement this?"
- "What's blocking quick action?"
- "What can you do this week?"

### Common Pitfalls to Avoid

⚠️ **Metric gaming** - Optimizing numbers without real improvement
- Solution: "What's the real goal behind this metric?"

⚠️ **Losing strategic focus** - Optimizing wrong things efficiently
- Solution: "Is this bottleneck actually important strategically?"

⚠️ **Over-optimizing current state** - Making today better instead of transforming
- Solution: "Are you optimizing the right thing, or is the whole approach wrong?"

⚠️ **Ignoring people/culture** - Treating humans as cogs
- Solution: "How do people feel about this? What resistance might you face?"

### Workshop Flow Example: Theory of Constraints

**Setup (5 min):**
- "We're going to find the ONE constraint limiting your whole system"
- "Then figure out how to exploit, elevate, or eliminate it"

**Introduction (5 min):**
- Eliyahu Goldratt, "The Goal"
- System is only as strong as weakest link
- Example: "If your sales process is limited by demo calls, optimizing website won't help"

**Application (30 min):**
1. **Map the system** (8 min)
   - What's the process/system?
   - What are the steps?
   - What's the goal/output?
   
2. **Identify constraint** (7 min)
   - Where's the bottleneck?
   - Where do things pile up/slow down?
   - What's the limiting factor?
   - How do you know it's the real constraint?
   
3. **Exploit** (5 min)
   - How to get most from current constraint?
   - Quick wins to improve bottleneck?
   
4. **Subordinate** (5 min)
   - How to align rest of system to constraint?
   - What should run at constraint's pace?
   
5. **Elevate** (5 min)
   - How to permanently fix constraint?
   - What investment needed?
   - What's next constraint after this one?

**Synthesis (8 min):**
- Primary constraint identified
- Quick wins (exploit)
- Long-term solution (elevate)
- Expected impact with metrics
- Implementation plan

## Cross-Framework Facilitation Principles

### Regardless of Framework Type:

**1. Start with Why**
- Why this framework? Why now?
- Connect framework to user's real problem

**2. Explain Before Applying**
- Don't jump into questions
- Give user mental model first

**3. Follow the Structure**
- Frameworks have phases for a reason
- Don't skip or reorder without good reason

**4. Adapt Depth to Engagement**
- If user is deeply engaged, go deeper
- If struggling, provide more examples/structure

**5. Make It Concrete**
- Ground everything in user's specific situation
- Abstract principles are useless without application

**6. Connect Insights Across Phases**
- "This relates to what you said earlier about..."
- "I'm seeing a pattern of..."

**7. Flag Pitfalls Proactively**
- Don't wait for user to fall into trap
- Warn early: "Common mistake here is..."

**8. End with Action**
- Every workshop ends with concrete next steps
- "What will you do based on this?"

**9. Document Thoroughly**
- Create complete workshop record
- User should be able to reference later

**10. Celebrate Insights**
- When user has breakthrough, acknowledge it
- "That's a key insight..."
- Makes workshop feel valuable

## Facilitation Anti-Patterns

### Don't:

❌ **Lecture** - This is workshop, not classroom
- Do: Ask questions that lead user to insights

❌ **Accept surface answers** - "That sounds good" without depth
- Do: Probe: "Can you be more specific?"

❌ **Rush through phases** - To "finish" framework
- Do: Better to go deep on fewer phases than shallow on all

❌ **Show off knowledge** - Citing framework trivia
- Do: Focus on what's useful for user

❌ **Ignore warning signs** - User is confused/frustrated
- Do: Pause, check in, adjust approach

❌ **Force fit** - Making framework work when it doesn't
- Do: Acknowledge limitations, adapt or suggest different framework

❌ **Skip documentation** - "I'll remember the insights"
- Do: Document as you go, create comprehensive record

❌ **End without action** - Just insights, no next steps
- Do: Always end with concrete actions

## Remember

**Your role is facilitator, not expert:**
- User knows their situation
- You know the framework
- Together you create insights neither could alone

**Frameworks are tools, not truth:**
- They simplify reality to make it manageable
- They have limitations
- Adapt when needed

**Value is in application, not theory:**
- User doesn't need to master framework
- User needs insights and actions
- Focus on useful, not comprehensive

**Different frameworks need different styles:**
- Strategic: Analytical and thorough
- Mental Models: Philosophical and probing
- Decision: Pragmatic and action-focused
- Innovation: Creative and exploratory
- Operational: Practical and metric-driven

**Adapt to your user:**
- Analytical users: More depth
- Action-oriented users: Faster to recommendations
- Struggling users: More examples and structure
- Expert users: Challenge more, explain less
