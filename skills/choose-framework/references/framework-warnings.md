# Framework Warnings & Mismatches

This guide identifies warning signs that a framework might be mismatched to the situation. Use these to avoid recommending frameworks that won't work or will frustrate users.

**Warning Levels:**
- 🔴 **FATAL** - Don't recommend this framework, it will fail
- 🟡 **SERIOUS** - Major concerns, warn user explicitly
- 🟠 **CAUTION** - Minor concerns, mention but may still work

---

## General Warning Signs

### Framework Too Complex for Situation

🔴 **FATAL when:**
- Using organizational frameworks (McKinsey 7S) for personal decisions
- Using multi-month frameworks for urgent decisions needed today
- Requiring data analysis when no data exists and can't be obtained

🟡 **SERIOUS when:**
- Framework requires team when user works alone
- Framework needs weeks when user has days
- Framework requires expertise user doesn't have

🟠 **CAUTION when:**
- Framework has many steps but some can be abbreviated
- Framework is complete but subset would work
- Learning curve exists but user is motivated

**How to handle:**
- Suggest simpler alternative
- Explain why complexity is a mismatch
- "This framework is designed for [complex situation], but your situation is [simpler]. Try [alternative] instead."

### Framework Too Simple for Situation

🔴 **FATAL when:**
- Using personal frameworks (Eisenhower Matrix) for organizational transformation
- Using quick frameworks (Pareto) for deep strategic analysis
- Using single-perspective frameworks for multi-stakeholder problems

🟡 **SERIOUS when:**
- Framework ignores key stakeholders
- Framework doesn't address complexity level
- Framework provides clarity but not depth needed

🟠 **CAUTION when:**
- Framework is starting point but needs follow-up
- Framework covers part but not all of problem
- Framework is useful but incomplete

**How to handle:**
- Suggest more comprehensive framework
- Or suggest sequence: simple framework first, then deeper one
- "This framework will help with [aspect], but for full picture you'll also need [additional framework]"

## Problem Type Mismatches

### Using Strategic Frameworks for Operational Problems

🔴 **FATAL:**
- Porter's Five Forces to optimize internal process
- Blue Ocean Strategy to fix broken workflow
- Wardley Mapping to prioritize today's tasks

🟡 **SERIOUS:**
- SWOT Analysis for process improvement (too general)
- Scenario Planning for operational decision (too abstract)

**Why it fails:**
Strategic frameworks analyze position and direction, not execution and optimization.

**Alternative:** Theory of Constraints, Pareto Principle, OODA Loop, Systems Thinking

### Using Operational Frameworks for Strategic Problems

🔴 **FATAL:**
- Pareto Principle for competitive positioning
- Eisenhower Matrix for business model design
- OKR Framework for market entry strategy

🟡 **SERIOUS:**
- Theory of Constraints for innovation (too focused on existing)
- OODA Loop for long-term planning (too tactical)

**Why it fails:**
Operational frameworks optimize execution, not strategy or direction.

**Alternative:** Porter's Five Forces, Blue Ocean, Wardley Mapping, Scenario Planning

### Using Innovation Frameworks for Optimization Problems

🔴 **FATAL:**
- Design Thinking to optimize existing process
- Lean Startup to scale proven business model
- Jobs-to-be-Done to improve efficiency

🟡 **SERIOUS:**
- First Principles for incremental improvement (overkill)
- Six Thinking Hats for operational decision (too exploratory)

**Why it fails:**
Innovation frameworks assume unknowns and need for discovery. Optimization has knowns.

**Alternative:** Pareto Principle, Theory of Constraints, OODA Loop

### Using Decision Frameworks for Analysis Problems

🔴 **FATAL:**
- Regret Minimization for competitive analysis
- Pre-Mortem for understanding market dynamics
- Inversion for mapping system complexity

🟡 **SERIOUS:**
- Eisenhower Matrix for strategic planning (too reductive)
- Second-Order Thinking for innovation (better for consequences)

**Why it fails:**
Decision frameworks help make choices, not understand situations.

**Alternative:** Porter's Five Forces, SWOT, Cynefin, Systems Thinking

## Context Mismatches

### Using Stable-Context Frameworks in Chaos/Uncertainty

🔴 **FATAL:**
- Detailed long-term planning in chaotic environment
- SWOT Analysis when situation changes daily
- Porter's Five Forces when industry is being disrupted

🟡 **SERIOUS:**
- Business Model Canvas in extreme uncertainty (too assumptive)
- BCG Matrix during market disruption (historical data misleading)

**Why it fails:**
These frameworks assume predictability and stability that doesn't exist.

**Alternative:** OODA Loop, Lean Startup, Scenario Planning, Cynefin Framework

### Using Complexity Frameworks for Simple Problems

🔴 **FATAL:**
- Systems Thinking for straightforward decision
- Cynefin Framework when cause-effect is obvious
- Mental Models latticework for clear yes/no decision

🟡 **SERIOUS:**
- First Principles for minor improvement (overkill)
- Wardley Mapping for tactical choice (too strategic)

**Why it fails:**
Complexity frameworks add analytical burden that simple problems don't need.

**Alternative:** Eisenhower Matrix, Pareto Principle, simple pros/cons list

### Using Fast-Cycle Frameworks for Deep Strategic Problems

🔴 **FATAL:**
- OODA Loop for life-changing decision
- Eisenhower Matrix for business model design
- Pareto Principle for market positioning

🟡 **SERIOUS:**
- RICE Framework for strategic direction (too tactical)
- Six Thinking Hats for deep analysis (good for ideation, not strategy)

**Why it fails:**
Fast frameworks prioritize speed over depth, wrong for strategic decisions.

**Alternative:** Regret Minimization, Scenario Planning, Porter's Five Forces

---

## Time Horizon Mismatches

### Long-Cycle Frameworks for Urgent Decisions

🔴 **FATAL when decision needed today/this week:**
- Scenario Planning (requires weeks/months)
- Design Thinking full process (needs iteration time)
- Wardley Mapping complete analysis (requires deep analysis)

🟡 **SERIOUS when decision needed this month:**
- McKinsey 7S transformation (multi-month process)
- Lean Startup full cycle (requires build-measure-learn)

🟠 **CAUTION:**
- Can abbreviated version work?
- Is urgency real or perceived?
- Can decision be delayed slightly?

**Alternative:** OODA Loop, Eisenhower Matrix, Pre-Mortem (quick version), Pareto

### Short-Cycle Frameworks for Strategic Decisions

🔴 **FATAL for life-changing/irreversible decisions:**
- Eisenhower Matrix for career pivot
- Pareto Principle for business model choice
- RICE Framework for strategic direction

🟡 **SERIOUS for important decisions:**
- OODA Loop for long-term positioning
- Quick prioritization for strategic investment

**Why it fails:**
Strategic decisions need depth and long-term thinking, not quick triage.

**Alternative:** Regret Minimization, Second-Order Thinking, Scenario Planning

---

## Data Availability Mismatches

### Data-Hungry Frameworks Without Data

🔴 **FATAL when no data available and can't get it:**
- Porter's Five Forces without industry data
- BCG Matrix without market growth data
- Risk Management Frameworks without risk data
- RICE Framework without Reach/Impact estimates

🟡 **SERIOUS when mostly assumptions:**
- SWOT with more opinions than facts
- Business Model Canvas with unvalidated assumptions
- OKR without baseline metrics

**Why it fails:**
These frameworks depend on data analysis. Garbage in, garbage out.

**Alternative:** First Principles, Lean Startup, Pre-Mortem, Design Thinking (discovery first)

**Exception:** If framework purpose is to identify what data to gather, it can work as planning tool.

### Experimentation Frameworks Without Ability to Experiment

🔴 **FATAL when can't iterate:**
- Lean Startup in regulated industry with 2-year approval cycles
- Design Thinking when can't prototype and test
- OODA Loop when can't act quickly

🟡 **SERIOUS when experimentation is hard:**
- Jobs-to-be-Done without customer access
- Rapid prototyping with high-cost prototypes

**Why it fails:**
These frameworks assume build-test-learn cycles that situation doesn't allow.

**Alternative:** Scenario Planning, Pre-Mortem, Inversion, Mental Models

## Stakeholder Mismatches

### Individual Frameworks for Organizational Problems

🔴 **FATAL:**
- Regret Minimization for company strategy (personal framework)
- Circle of Competence for organizational decision (individual lens)
- Personal prioritization for team problem

🟡 **SERIOUS:**
- First Principles for change requiring buy-in (great thinking, but what about others?)
- Second-Order Thinking for political decisions (ignores stakeholders)

**Why it fails:**
Individual frameworks ignore organizational dynamics, politics, and multiple perspectives.

**Alternative:** McKinsey 7S, Systems Thinking, Six Thinking Hats, Scenario Planning

### Organizational Frameworks for Personal Decisions

🔴 **FATAL:**
- McKinsey 7S for career choice (massive overkill)
- Wardley Mapping for personal planning (wrong scale)
- Porter's Five Forces for life decisions (weird application)

🟡 **SERIOUS:**
- OKR for personal goals (can work but usually overkill)
- SWOT for personal decision (more useful for external analysis)

**Why it fails:**
Organizational frameworks assume complexity, stakeholders, and scale that don't apply personally.

**Alternative:** Regret Minimization, Circle of Competence, Second-Order Thinking

### Consensus Frameworks Without Authority

🔴 **FATAL when user lacks authority:**
- Six Thinking Hats requiring full team
- McKinsey 7S requiring org-wide involvement
- Design Thinking requiring cross-functional collaboration

🟡 **SERIOUS:**
- OKR requiring executive sponsorship
- Theory of Constraints requiring process ownership

**Why it fails:**
User can't convene stakeholders or implement findings.

**Alternative:** Analyze individually with frameworks you can apply alone, then present findings

## Implementation Complexity Mismatches

### High-Complexity Frameworks for Low-Capacity Situations

🔴 **FATAL:**
- Multi-month framework when user has 2 hours
- Team-based framework when user is alone
- Resource-intensive framework with no budget

🟡 **SERIOUS:**
- Design Thinking full process with limited time
- Wardley Mapping without strategic planning capacity

**How to handle:**
- Suggest abbreviated version
- Suggest simpler alternative
- Explain time/resource requirements clearly

### Simple Frameworks for Complex Implementation

🔴 **FATAL:**
- Eisenhower Matrix for organizational transformation
- Pareto Principle for complex system change
- Simple prioritization for political environment

🟡 **SERIOUS:**
- Basic frameworks for change requiring extensive buy-in
- Individual frameworks for system-wide problems

**Why it fails:**
Implementation complexity needs frameworks that address organizational dynamics.

**Alternative:** McKinsey 7S, Systems Thinking, Scenario Planning

---

## User Capability Mismatches

### Advanced Frameworks for Beginners

🟡 **SERIOUS:**
- Wardley Mapping as first framework ever (steep learning curve)
- Systems Thinking without training (conceptually difficult)
- Mental Models latticework without foundation (too abstract)

🟠 **CAUTION:**
- Any framework requiring deep expertise user lacks
- Frameworks assuming background knowledge

**How to handle:**
- Assess user experience level
- Offer to explain framework thoroughly
- Suggest starting with simpler framework
- Provide extra guidance and examples

### Basic Frameworks for Sophisticated Users

🟠 **CAUTION:**
- Eisenhower Matrix for experienced strategist (may feel too simple)
- Basic SWOT for expert analyst (may need more sophisticated approach)

**How to handle:**
- User may still benefit from basics
- Or suggest advanced alternative
- Or use simple framework but go very deep

## Common Combination Mistakes

### Frameworks That Don't Sequence Well

🟡 **SERIOUS:**
- Lean Startup → Scenario Planning (contradictory - experiment vs plan)
- Regret Minimization → OODA Loop (long-term vs short-term)
- Design Thinking → Theory of Constraints (innovation vs optimization)

### Frameworks That Sequence Perfectly

✅ **GOOD COMBINATIONS:**
- Choose-Framework → Use-Framework (this is the designed flow!)
- Porter's Five Forces → Blue Ocean Strategy (analyze then position)
- Jobs-to-be-Done → Design Thinking (understand then create)
- Cynefin → appropriate framework for that domain (assess then apply)
- Regret Minimization → First Principles (decide then figure out how)

## Red Flag Phrases from Users

Watch for these phrases that signal potential mismatches:

### "I need to decide today"
⚠️ Don't recommend: Design Thinking, Scenario Planning, Lean Startup
✅ Recommend: OODA Loop, Eisenhower Matrix, Pre-Mortem (quick)

### "I don't have any data"
⚠️ Don't recommend: Porter's Five Forces, BCG Matrix, RICE, Risk Management
✅ Recommend: Lean Startup, First Principles, Inversion, Pre-Mortem

### "I'm just one person"
⚠️ Don't recommend: McKinsey 7S, Six Thinking Hats (full), Wardley Mapping (org-wide)
✅ Recommend: Regret Minimization, Circle of Competence, Second-Order Thinking

### "This affects the whole company"
⚠️ Don't recommend: Personal frameworks, quick tactical frameworks
✅ Recommend: McKinsey 7S, Wardley Mapping, Scenario Planning, Systems Thinking

### "Everything is changing constantly"
⚠️ Don't recommend: Long-term planning, stable-context frameworks
✅ Recommend: OODA Loop, Lean Startup, Cynefin, Scenario Planning

### "I have no idea where to start"
⚠️ Don't recommend: Advanced frameworks, frameworks assuming clarity
✅ Recommend: First Principles, Cynefin (sense-making), Mental Models

## How to Deliver Warnings

### When Framework is Wrong

**Don't:**
❌ "That framework won't work for you"
❌ "You picked the wrong framework"
❌ Just ignore user's suggestion

**Do:**
✅ "That framework could work, but [Framework X] might be better because [reasons specific to their situation]"
✅ "Let me understand your situation first to make sure [Framework] is the best fit"
✅ "[Framework] is great for [X], but your situation is more about [Y]. How about [Alternative]?"

### When Framework Has Issues

**Format:**
"[Framework] fits in these ways: [positives]

However, I want to flag: [concern] because [reason].

We could work around this by [solution], or consider [alternative] which handles [concern] better.

What do you think?"

### When User Insists on Wrong Framework

**Don't fight it.** Validate and redirect:

"I hear that [Framework] resonates with you. Let's make sure it's the right fit by understanding your situation first. [Ask key questions]

[After questions, if still wrong:]

Based on what you've shared, [Framework] could work for [aspects], but I'm concerned about [specific mismatch]. [Alternative] might handle [aspect] better.

Would you like to try [Alternative], or would you prefer to proceed with [Framework] and we'll adapt it?"

Give user choice, but informed choice.

---

## Remember

1. **Warn early** - Don't wait until framework fails mid-workshop
2. **Be specific** - Don't just say "bad fit," explain exactly why
3. **Offer alternatives** - Never just critique, always suggest better option
4. **Explain reasoning** - Connect warning to user's specific situation
5. **Let user choose** - If they insist despite warning, proceed but adapt
6. **No perfect framework** - Even good matches have limitations, that's okay

**Goal:** Help user avoid frustration, not show off framework knowledge. Best recommendation is the one that works for them.
